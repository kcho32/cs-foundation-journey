# Array (Draft)
Array는 저장 방식
List는 순서가 있는 데이터 모음 -> 약속/규격

장점:
빠른 조회 속도
인덱스 -> 메모리 주소 계산이므로 인덱스로 바로 접근 가능 -> O(1)의 속도

원래 Array는 공간이 정해져 있어서 해당 갯수만큼의 데이터를 저장할 수 있음 (크기 고정)

많은 언어에서 자주 쓰는 List 구현체는 Array 기반 Dynamic Array인 경우가 많다. 예를 들어 Java의 ArrayList, Python의 list, JavaScript의 Array가 그렇다. 하지만 List라는 개념 자체가 항상 Array 기반인 것은 아니다. (LinkedList는 Node 기반)

크기가 고정된 Array와 달리 Dynamic한 사이즈를 가지고 있음
Array기반으로 고정된 목록 만든 후, 사이즈를 넘어서 데이터가 들어오면 더 큰 크기의 Array를 만들고 값을 복사 -> 모두 복사를 한번 해야하기 때문에 O(n)의 속도

데이터를 뒤에 붙이는 것은 빈 자리에 있는 Array라면 바로 뒤에 붙이므로 O(1)이지만 공간 부족으로 새로운 배열을 만들고 복사해야하면 O(n). 평균이 O(1)이므로 Amortized O(1)

앞이나 중간에 붙이는 것은 뒤에 붙이는 것보다 느림 -> 값을 넣은 후 뒤 값들을 뒤로 밀어야하기 때문에 (삭제도 마찬가지) -> O(n) 

값이 어딨는 지를 검색하는 것은 처음부터 끝까지 뒤져야하므로 느릴 수 있음 -> O(n)
만약 데이터가 정렬되어 있다면 이분 탐색으로 O(log n)으로 줄일 수 있다

수 많은 자료구조들이 Array를 기반으로 둔다.

Array는 값들이 메모리에 저장될 때 연속적으로 붙어서 저장되기 때문에 CPU 캐시가 효율적으로 동작 (메모리 지역성이 좋다)

우리가 코드에서 사용하느 List(ArrayList)는 결국 내부적으로 Array를 사용하는 동적 자료구조


LinkedList
값 하나가 노드로 구현되어서 다음 노드와 연결되는 자료구조

LinkedList는 Array와 다르게 노드 위치만 알고 있으면 삽입 및 삭제가 빠름 -> 노드 삭제후 이전, 뒤 노드를 이어주면 되기 때문 -> O(1)
인덱스나 값으로 찾아야한다면 마찬가지로 찾는데 -> O(n)

크기가 정해져 있지 않아 추가 및 줄이는데 용이함

각 노드는 메모리 여기 저기에 흩어져 있을 수 있기 때문에 캐시 효율성이 좋지는 않음
실제로 논리적으로는 연결되어 있어도 메모리에는 Array처럼 연속으로 이어져 있지 않을 가능성이 높아서 조회에 오래걸릴 수 있음.

예를 들어 순회는 Array와 같은 O(n)이지만 캐시 효율성으로 인해 Array가 더 빠름
노드를 읽고 다른 주소로 이동 -> pointer chasing

또한 인덱스로 주소 계산이 바로 안되기 때문에 100번 째 노드를 찾기 위해서는 head(1번)부터 next로 100번을 따라가야함 -> O(n)

LinkedList는 값 이외의 연결 정보도 저장하기 때문에 추가 메모리 필요

그래서 실제로 LinkedList를 사용하는 경우는 잘 없음 - Queue 용도도 Deque를 많이 사용하는 편'


Multi-dimensianl Array
개념적으로는 배열 안의 배열이다.
C에서는 2차원 배열이 연속된 하나의 큰 배열로 저장

Python의 list of lists:
바깥 리스트가 안쪽 리스트들의 참조를 저장한다.

Java의 int[][]:
바깥 배열이 안쪽 int[] 배열들의 참조를 저장한다.

C의 int arr[2][3]:
하나의 연속된 메모리 블록에 row-major 방식으로 저장된다.
메모리에 저장은 1차로 되서 메모리에서 1차 2차 배열은 알 수 없다.
선언할 때의 크기 int arr[2][3] 이걸로 row by column을 계산하는 것
